# Momentum Resolution pipeline (summary)

This project estimates track momentum resolution by propagating a state through a sequence of **Planes** (measurement planes and passive material planes), building transport matrices and process noise (multiple scattering), and then writing per-track results to a ROOT `TNtuple`.

## Main entry point

- Runner: `src/Apps/run_sampled_tracks.py`
  - Loads settings from a YAML file (example: `layouts/u2_debug_material.yaml`).
  - Reads kinematics from the `--kinematic_file` parquet (default: `samples/bsmumu_rdf.parquet`).
  - For each track state (`M1` and `M2`) calls `Core.MomentumResolutionCalculator.ComputeResolutions(...)`.
  - Writes a ROOT file with an `ntuple` tree (TNtuple).

## Inputs and how they are used

### Settings (YAML)

The runner expects a YAML file with at least:

- `detectionlayers`: list of CSV files describing planes
- `fieldMap`: CSV field map (grid)
- `gdmlInput`: GDML file or ROOT geometry file readable by `TGeoManager.Import`
- `storing_planes`: CSV describing extra planes where state/covariance is stored
- `fieldShrinkFactor`, `fieldMagFactor`: scale factors for the field grid and vector magnitude
- `skipVelo`, `skipRICH1`, `skipUT`, `skipFT`, `skipWorld`: toggles that affect what geometry material is used
- `VeloMeasurementsFromGDML`, `UPMeasurementsFromGDML`, `DownMeasurementsFromGDML`: toggles for creating measurement planes from GDML nodes

- **Detector parameterization (CSV)**
  - `settings["detectionlayers"]` is a list of CSV files.
  - Loaded in `Core/MomentumResolutionCalculator.py` and filtered by `Use == 1`.
  - Each row becomes a `Core.Plane.Plane` with:
    - `Z`, `thickness` (interpreted as X/X0), optional `sigmaX/sigmaY` and `angle`, `xyshape`, `hiteff`.

- **B-field (CSV map)**
  - Loaded by `Core/BField.py` from `settings["fieldMap"]`.
  - Converted into three ROOT `TH3D` histograms (`Bx`, `By`, `Bz`) and queried via trilinear interpolation.
  - Used by the `Core/ParabolicExtrapolator.py` trajectory creation.

- **Geometry/material navigation (GDML / ROOT TGeo)**
  - Loaded by `Core/MaterialLocator.py` from `settings["gdmlInput"]`.
  - Used to retrieve volumes crossed between successive states/planes and to inject passive material planes (and optionally measurement planes) derived from geometry.

- **“Storing planes” (CSV)**
  - Optional `settings["storing_planes"]`.
  - Defines additional planes where the runner stores propagated state and covariance summaries.

## Outputs

- Output file is a ROOT file containing:
  - `TNtuple` named `ntuple`
  - Baseline branches (`charge`, `p`, `pt`, `eta`, `phi`, material summary counters, etc.)
  - For each named plane group:
    - state at plane: `x_*`, `y_*`, `z_*`, `tx_*`, `ty_*`
    - forward-fit covariance summaries: `fwd_sigmaX_*`, `fwd_sigmaY_*`, `fwd_sigmaTX_*`, `fwd_sigmaTY_*`, `fwd_pres_*`

The exact list is generated by `MomentumResolutionCalculator.GetFillingBranches(...)`.

The runner writes to `--output_file` but internally appends `_{start_entry}_{stop_entry}` before `.root` so parallel chunks can be merged/compared later.
